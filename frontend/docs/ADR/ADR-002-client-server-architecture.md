# ADR-002: Client-Server 資料架構

- **狀態**: Proposed
- **日期**: 2026-06-18
- **關聯**: [ADR-001](ADR-001-app-user-flow.md), [ADR-003](ADR-003-ai-food-recognition-pipeline.md)

## Context

食誌 App 需要同時支援離線飲食記錄（無網路時仍可使用）與線上功能（AI 辨識、通用食物資料庫查詢）。需要決定哪些資料存在 Client 端、哪些存在 Server 端，以及兩端的技術選型。

## Decision

### 技術選型

| 層級 | 選擇 | 備註 |
|------|------|------|
| **Client** | Flutter + SQLite | Android 為主要平台 |
| **Server** | Node.js | 團隊熟悉的技術棧 |
| **部署** | Docker / Cloudflare 生態系 | 控制在免費額度內 |
| **CI/CD** | CodeMagic + Firebase Distribution | Android 建置與測試分發 |

### 資料職責劃分

```
┌─────────────────────────────┐     ┌─────────────────────────────┐
│     Flutter Client (SQLite) │     │       Server (Node.js)      │
├─────────────────────────────┤     ├─────────────────────────────┤
│  · 飲食記錄（餐點、品項）     │     │  · 食物資料庫（多來源）       │
│  · 常吃食物清單              │     │    OFF / FatSecret / TFDA   │
│  · 自定義食物                │     │  · AI 辨識服務               │
│  · 精簡食物資料庫（離線搜尋） │     │  · 食物圖片儲存              │
│  · 身體數據（體重、身高）     │     │  · 登入驗證 / API 授權       │
│  · 每日目標與偏好設定         │     │                             │
└──────────────┬──────────────┘     └──────────────┬──────────────┘
               │          REST API / HTTPS          │
               └────────────────────────────────────┘
```

#### Client 端（Single Source of Truth）

| 資料 | 說明 |
|------|------|
| 飲食記錄 | 每日餐點、食物品項、份量、時間、熱量與三大營養素 |
| 常吃食物清單 | 依據使用者記錄頻率自動產生，加速未來記錄 |
| 自定義食物 | 使用者自行建立的食物與營養資訊 |
| 精簡食物資料庫 | Onboarding 時從 Server 下載，提供離線食物名稱搜尋 |
| 身體數據 | 身高、每日體重紀錄、目標體重、目標日期 |
| 目標與設定 | 每日熱量目標、偏好設定 |

#### Server 端（Single Source of Truth）

| 資料 | 說明 |
|------|------|
| 食物資料庫 | 整合多來源，統一提供搜尋與條碼查詢 API |
| AI 辨識服務 | 接收食物圖片，回傳辨識結果（詳見 [ADR-003](ADR-003-ai-food-recognition-pipeline.md)） |
| 食物圖片儲存 | 使用者上傳的食物照片 |
| 登入驗證 | 帳號認證與 API Token 管理，防止 API 濫用 |

### 食物資料庫來源

目標受眾為台灣使用者，整合以下三個來源：

| 來源 | 適用情境 | 條碼 | 授權 |
|------|----------|------|------|
| **Open Food Facts (OFF)** | 包裝食品條碼查詢、掃描營養標示擴充 | ✅ EAN-13 | 開源免費 |
| **FatSecret API** | 台灣超商/包裝食品，條碼覆蓋率最高 | ✅ EAN-13 | 商業（免費 Tier 有次數限制） |
| **TFDA 食品營養成分資料庫** | 原型食物（蔬菜、肉類等），AI 辨識後查營養素 | ❌ | 政府公開資料 |

查詢策略概要：
- **條碼掃描**：優先查 OFF（免費）→ fallback FatSecret（覆蓋率高但有額度）→ 查無資料時引導拍攝營養標示
- **AI 辨識 / 名稱搜尋**：優先查 TFDA（原型食物最權威）→ fallback OFF
- **離線搜尋**：查詢本地精簡資料庫

#### 資料來源標注

所有回傳結果標注 `source` 欄位：

| source | 來源 |
|--------|------|
| `off` | Open Food Facts |
| `fatsecret` | FatSecret API |
| `tfda` | TFDA 食品營養成分資料庫 |
| `user` | 使用者自定義食物 |

#### 本地精簡資料庫

Onboarding 時從 Server 下載一份精簡版食物資料庫到本地 SQLite：

- **來源**：以 TFDA 為主（台灣常見原型食物），輔以 OFF 高頻包裝食品
- **內容**：食物名稱、每 100g 營養素、常用份量單位（不含圖片與條碼）
- **更新**：App 啟動時背景檢查版本號，有新版本時提示更新

### 主要 API

| 端點 | 方法 | 說明 |
|------|------|------|
| `/api/auth/login` | POST | 登入驗證，回傳 API Token |
| `/api/food/search` | GET | 食物名稱搜尋（整合 TFDA + OFF） |
| `/api/food/barcode/:code` | GET | 條碼查詢（整合 OFF + FatSecret） |
| `/api/recognize` | POST | 上傳食物圖片，回傳 AI 辨識結果 |
| `/api/food-db/version` | GET | 精簡資料庫版本檢查 |
| `/api/food-db/download` | GET | 下載精簡資料庫 |

## Consequences

### 優點

- **離線可用**：飲食記錄、體重紀錄、統計查看、食物名稱搜尋不依賴網路
- **隱私保護**：個人飲食與身體數據留在本地裝置
- **降低 Server 負載**：Server 僅處理 AI 辨識與食物資料庫查詢
- **低延遲**：本地 SQLite 讀寫即時

### 缺點

- **跨裝置不同步**：換機時本地資料無法自動遷移（未來可考慮匯出/匯入）
- **離線限制**：AI 辨識與條碼掃描仍需網路

### 待決定事項

- [ ] 登入驗證方式（Email + Password / OAuth / Sign in with Apple）
- [ ] API Rate Limiting 策略
- [ ] SQLite schema 設計
- [ ] FatSecret API 是否導入（取決於免費額度是否足夠初期）
- [ ] 精簡資料庫的收錄範圍與大小上限
- [ ] 精簡資料庫版本管理（全量 vs 增量更新）

> **備註**：查詢策略細節、快取設計、部署配置等實作層面內容，分別參見前後端各自的技術文件。
